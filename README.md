# SPEngine (название в процессе смены)

Минималистичный, хардкорный и полностью статический ECS-игровой движок для оригинальной PlayStation 1 (PS1). Написан на C++ на базе **PSn00bSDK**.

Движок создавался не как одноразовая демка для портфолио, а как **полноценный низкоуровневый каркас для разработки реальных игр**. Здесь нет визуального редактора и тяжелого ООП. Ваш код — это и есть ваш редактор. Движок сознательно спроекпрован без рантайм-защиты от дурака: любая критическая ошибка, выход за пределы массива или неверный адрес памяти мгновенно вешают консоль, что гарантирует максимальный перформанс железа 1994 года без абстрактного оверхеда.

---

## 💾 1. Философия памяти и Системный Базис

У PlayStation 1 всего **2 МБ общей оперативной памяти (RAM)**. Главный враг стабильности на таком железе — фрагментация памяти при постоянных аллокациях. SPEngine решает эту проблему радикально: **полный отказ от динамической памяти (`malloc` / `free`) во время игрового процесса**.

*   **Принцип «Столько хватит»:** Все внутренние структуры движка, массивы компонентов и сущностей имеют фиксированные жесткие лимиты, заданные на этапе компиляции. Оптимизация объема памяти под конкретную игру выполняется вручную разработчиком: если памяти много — размеры массивов уменьшаются в коде, если не хватает — лимиты повышаются.
*   **Память ядра:** В базовой конфигурации (включая служебные выравнивания, текстуры, геометрию куба и тестовые сцены) движок потребляет всего **178 754 байта** (~175 КБ) RAM, отдавая весь остальной объем под нужды разработчика.

## 🛠 2. Полный API-Справочник Подсистем

### 🧬 Реестр Сущностей (`Registry`)
Компоненты сущностей развернуты в плоские массивы (Structure of Arrays) через макрос `COMPONENT_LIST`. ID сущности — это прямой индекс в массивах компонентов текущей сцены. 

```cpp
#define COMPONENT_LIST \
    X(Position, positions) \
    X(Size, sizes) \
    X(Rotation, rotations) \
    X(Color, colors) \
    X(Mesh, meshes) \
    X(Texture, textures)
```

*   `int Registry::CreateEntity()` — Находит первый свободный ID путем линейного перебора массива до `MAX_OBJECTS`. Проверяет флаг `ACTIVE`. Если индекс свободен, взводит флаг `ACTIVE` и возвращает ID. Ноль оверхеда на аллокацию. Если лимит превышен, возвращает `-1`.
*   `void Registry::DestroyEntity(int id)` — Мгновенно освобождает ID, сбрасывая маску флагов `flags[id] = 0`. Данные в массивах компонентов не затираются, экономя такты процессора — они просто перезапишутся при создании новой сущности на этом индексе.

### 🕹 Модуль Ввода (`Pad`)
Низкоуровневая абстракция над цифровыми портами геймпада PS1. Работает через инвертирование стандартных масок SDK.
*   `int Pad::Initialize()` — Инициализирует шину геймпада через `InitPAD` и `StartPAD` с буферами по 34 байта. Заполняет массив `bindings` для маппинга масок кнопок SDK (например, `PAD_START`) на внутренние логические флаги движка (`START`, `CANCEL`, `UP`, `DOWN`, `USE`, `ACTION`).
*   `void Pad::Update()` — Опрашивает состояние контроллера. Если геймпад готов (`!pad->stat`), функция сбрасывает общую маску состояний `Pad::states = 0`, пробегает по биндингам и через `!(pad->btn & bindings[i].key)` проверяет нажатия. Результат собирается в единую битовую маску `Pad::states`.
*   **Использование в игре:** Проверка нажатия выполняется через побитовое И: `if (Pad::states & UP) { ... }`.

### 💿 Асинхронный CD-Parser (`Parser`)
Подсистема для прямого низкоуровневого общения с приводом CD-ROM по стандарту ISO9660.
*   `int Parser::InitializeCD()` — Вызывает инициализацию дисковода через `CdInit()`.
*   `void Parser::ReadFile(const char* name)` — Запускает асинхронное чтение файла с диска. Метод автоматически форматирует имя файла под требования файловой системы PS1, превращая `"BALL.TIM"` в `"\\BALL.TIM;1"`. Ищет файл через `CdSearchFile`, переводит лазерную головку на физические сектора файла через `CdlSetloc` и инициирует чтение данных в статический буфер `Parser::buffer` на максимальной скорости привода (`CdlModeSpeed`). Метод завершает работу мгновенно, не дожидаясь окончания физического чтения.
*   `int Parser::CheckRead(uint8_t mode)` — Блокирующий или неблокирующий контроль чтения. Вызов `Parser::CheckRead(0)` выполняет синхронизацию с приводом через `CdReadSync(0, nullptr)`, останавливая выполнение потока до тех пор, пока лазер полностью не скопирует сектора файла в RAM-буфер. Вызов `Parser::CheckRead(0)`, соответственно, не останавливает поток, а возвращает 0 когда копирование готово и 1 когда не готово соответственно.
*   `uint32_t* Parser::LoadFile()` — Возвращает прямой указатель на статический буфер данных `Parser::buffer`.

### 📦 Хранилище Ресурсов с Дедупликацией (`Storage`)
Глобальный менеджер ресурсов для хранения бинарных данных (текстур, звуков) в RAM между сценами.
*   `int Storage::AddFile(uint16_t size, uint32_t* data)` — Регистрирует файл в таблице `Storage::files`. Перед аллокацией проверяет файлы через `memcmp`. Если файл с таким же размером и байтовым содержимым уже загружен, Storage **не выделяет новую память**, а просто возвращает ID уже существующего ресурса. Если ресурс уникален, размер выравнивается по границе 4 байт через побитовую операцию `(size + 3) & ~3`, выделяется память через `malloc`, данные копируются через `memcpy`, а индекс помечается в битовой маске `flags_files`.
*   `File* Storage::GetFile(uint8_t id)` — Возвращает указатель на структуру файла по его ID.
*   `void Storage::RemoveFile(uint8_t id)` — Освобождает память ресурса через `free` и сбрасывает бит в маске `flags_files`.
*   `int Storage::InitializeScratchpad()` — Настраивает прямую привязку сверхбыстрой локальной памяти процессора (Data Cache) через указатели прямо к физическому аппаратному адресу ядра `0x1F800000` для работы с критическими runtime-данными с нулевым оверхедом. (SCRATCHPAD НЕ ГОТОВ)

### 🎬 Режиссёр Эффектов (`Fx`)
Система бесшовных переходов между сценами. Метод `Fx::StartFX(Scene* ps, Scene* ns, uint8_t in, uint8_t out)` полностью перехватывает управление, замораживая стандартный игровой цикл. 
*   **Двухфазный цикл `Fx::Update()`:**
    1.  **Фаза Входа (`typeFxIn`):** Движок крутит цикл до значения `maxFrame / 2`. На каждом кадре вызывается логика *старой* сцены, но поверх неё отрабатывает функция визуального эффекта `Run[typeFx]()`, которая создаёт временные служебные сущности (например, прямоугольники для шахматной сетки) напрямую в текущем ECS-реестре старой сцены.
    2.  **Экватор перехода:** Ровно на середине анимации `Fx::Update` инициализирует *новую* сцену (`nextScene->Initialize()`), подменяет глобальный указатель текущей сцены `Game::currentScene = nextScene`, перепривязывает ECS-реестр новой сцены к системам `Render` и `Geometry`, а старую сцену полностью уничтожает через `delete prevScene`.
    3.  **Фаза Выхода (`typeFxOut`):** Движок докручивает кадры до `maxFrame`, вызывая логику эффекта уже внутри ECS-реестра *новой* сцены. По окончании эффекта служебные сущности удаляются, и управление возвращается в главный цикл.
*   **Доступные эффекты (`Fx::Run`):** `Fade` (плавное затемнение через блендинг `ABR_2`), `Swipe` (линейный сдвиг, не готов), `Chessboard` (анимация масштаба сетки блоков), `ChessSwipe` (прогрессивное появление блоков с побитовым расчётом задержки `((maxFrame / 3 * 179) >> 8)` на основе координат).

### 🔊 Аудио-Движок (`Sound`)
Низкоуровневая подсистема для прямого управления звуковым чипом SPU оригинальной консоли. 

```cpp
VAG      Sound::audios[512]; // Таблица дескрипторов звуков
uint16_t Sound::audiox;      // Текущий курсор аудио-аллокатора
```

*   `int Sound::Initialize()` — Инициализирует аудиоаппаратуру через `SpuInit()`, переводит передачу данных в режим DMA (`SPU_TRANSFER_BY_DMA`) для разгрузки CPU и выставляет мастер-громкость на максимум (`0x3FFF`). Первому элементу в таблице принудительно задаётся базовый начальный адрес звуковой памяти SPU RAM — `0x1010`.
*   `int Sound::SetAudio(uint32_t* data)` — Регистрирует `.VAG` файл (ADPCM), считанный с CD-ROM. Метод парсит заголовок файла, налету меняет порядок байт через макрос `SWAP_ENDIAN`, высчитывает точный размер аудиоданных с выравниванием под DMA-трансфер через побитовое И (`& 0xffffffc0`) и считывает частоту дискретизации (`rate`). Аудиоданные заливаются в SPU RAM со смещением от хедера (`data + 12`) с помощью `SpuWrite`. Метод автоматически рассчитывает адрес в звуковой памяти для следующего файла (`audio->data + audio->size`) и возвращает ID текущей дорожки.
*   `void Sound::Play(uint16_t id, uint8_t channel)` — Запускает воспроизведение звука на конкретном аппаратном канале SPU. Метод сбрасывает и взводит ключи канала через `SpuSetKey`, а также напрямую через макросы SDK настраивает ADSR-регистры чипа (`SPU_CH_FREQ`, `SPU_CH_ADDR`, громкость `0x3fff` и параметры огибающей `ADSR1 = 0x00ff` / `ADSR2 = 0x0000`).

## 📺 3. Графика, Геометрия и Конвейер Рендеринга

Движок жестко залочен под родное разрешение PS1 — **320x240 пикселей** (режим `MODE_NTSC`). Вывод графики использует классическую двойную буферизацию кадров (`DB db[2]`). Очистка экрана происходит аппаратно на уровне GPU: в `Render::Initialize()` взводится флаг `draw.isbg = 1`, за счёт чего видеопамять автоматически заливается фоновым серым цветом (`50, 50, 50`) при каждом вызове `Render::Present()`, полностью исключая эффект «шлейфа».

### Сортировка глубин и Ограничения (Ordering Table)
Структура `DB` тащит массив сортировки `uint32_t ot[LENGTH_OT]` размером аж в **4096 уровней**, что даёт высокую точность Z-сортировки примитивов. Внутри структуры зашиты жесткие лимиты — по **512 элементов** на каждый тип примитивов (`tiles`, `dr_tpages`, `tims`, `polys`). 
Если разработчик попытается отрендерить 513-й полигон куба или 513-ю текстуру, **движок гарантированно не упадёт в панику и не повесит консоль**: он просто молча проигнорирует лишние объекты, остановит сортировку для этого типа данных и продолжит отрисовку оставшегося кадра в `Present()`.

### 2D-Рендеринг и Полупрозрачность (`flip`)
Метод `flip()` распределяет 2D-объекты по таблице отрисовки. 
*   **Примитивы (`PRIMITIVE`):** Отрисовывает плоские полигоны `POLY_F4` по координатам `pos.vz`. Если включена прозрачность (`TRANS`), движок вытягивает режим блендинга через макрос `(flags & ABR_MASK) >> 6` (поддерживаются режимы `ABR_0`–`ABR_3`, включая вычитание цветов `ABR_2` для глубоких эффектов переходов) и инжектирует команду смены страницы `DR_TPAGE` прямо в конвейер перед полигоном.
*   **Текстуры:** Отрисовывает текстурированные полигоны `POLY_FT4`. Данные `.TIM` файла парсятся через `GetTimInfo` и заливаются во VRAM через `LoadImage`. Координаты текстурных страниц (`prect`) и палитры CLUT (`crect`) зашиваются в ECS-компонент `textures`. *Технический нюанс: компонента `Scale` в движке нет. На данный момент текстура выводится в своём исходном разрешении из файла, независимо от компонента `Size`.*

### 3D-Конвейер и Аппаратная Оптимизация (`Geometry::Update`)
Движок реализует полноценный 3D-рендеринг мешей силами геометрического сопроцессора GTE. 
1.  **Трансформация:** Метод `upd_cam()` инвертирует позицию камеры, считает матрицу вращения через `RotMatrix` и формирует итоговую матрицу вида. Для каждого 3D-объекта метод `set_matrix()` перемножает матрицы объекта и камеры через `CompMatrixLV`, отправляя результат напрямую в регистры GTE (`gte_SetRotMatrix`).
2.  **Аппаратный Бэкфейс-Куллинг:** Вершины треугольников загружаются в GTE через `gte_ldv3`, трансформируются через `gte_rtpt()` и обрезаются по экрану через `gte_nclip()`. Метод `gte_stopz(&p)` проверяет нормаль полигона: если треугольник повёрнут к камере спиной (`p <= 0`), он **мгновенно отбрасывается**, экономя такты CPU.
3.  **Z-Буферизация:** Для прошедших куллинг полигонов `POLY_F3` вычисляется средняя глубина треугольника через `gte_avsz3()`, значение `otz` валидируется строго в диапазоне `1 < otz < 4096`, после чего экранные координаты вершин забираются из регистров GTE через `gte_stsxy3` и полигон отправляется на отрисовку: `addPrim(&db[dbx].ot[otz], poly)`.

## 📅 4. Планы на будущее (Дорожная карта)

Развитие фреймворка распланировано в строгом хронологическом порядке:

1.  **Стриминг аудио (CD-DA):** Реализация воспроизведения фоновой музыки напрямую с дорожек игрового диска через парсер `src/parser/sound.cpp`. Лазер при этом физически блокируется под нужды аудиовещания, что исключает необходимость писать сложную и тяжелую логику прерываний привода. В перспективе возможна доработка до формата XA.
2.  **Полноценный Scratchpad:** Написание функционального API для практического использования сверхбыстрой локальной памяти процессора (Data Cache). На текущий момент реализована только жесткая ссылка на массив напрямую по физическому аппаратному адресу ядра `0x1F800000` объемом в 1 КБ (256 ячеек по 32 бита):
    ```cpp
    uint32_t (&Storage::scratchpad)[256] = *(uint32_t(*)[256])0x1F800000;
    ```
3.  **Сохранения (Memory Card):** Написание модуля `src/parser/memory_card.cpp` для работы со слотами карт памяти. Это избавит игры от структуры «длинного мультика» и позволит сохранять/загружать runtime-данные между сессиями.
4.  **Модернизация Рендера (Масштабирование):** Модификация метода `Render::flip()` — добавление автоматического апскейла/даунскейла текстур из VRAM до физических размеров, указанных в компоненте `Size`.
5.  **Глобальный архитектурный рефакторинг:**
    *   **Человечный API:** Создание более удобного высокоуровневого интерфейса для разработчика без потери ультимативной низкоуровневой эффективности кода.
    *   **Продвинутый Fx Оверлорд:** Отказ от обязательного деструкта старой сцены на экваторе перехода. Это позволит реализовать сцены-апплеты (например, меню паузы, открывающееся поверх геймплея), либо доработка API рендера для захвата и передачи текущего кадра экрана в качестве текстуры.
6.  **Расширение Ввода:** Добавление системы рантайм-перебинда клавиш в маске состояний и внедрение полноценной поддержки второго геймпада (Player 2).
7.  **Экспериментальный модульный движок (.SCN):** Архитектурная идея по компиляции игровых сцен в виде полностью изолированных исполняемых модулей. Движок будет подгружать файл сцены `name.scn` с диска через CD-парсер и динамически запускать его целиком, полностью освобождая RAM от кода предыдущих игровых зон (аналог оверлеев).
